# 🧸 Easy README – Kubernetes Commands (Pod Working Guide)

This README explains:

- What each command does
- How it works
- Why we use it (in very easy student-friendly way)

---

# 🚀 1. `kubectl apply -f .\grade-submission-portal-pod.yaml`

## 🌟 Simple Meaning

This command tells Kubernetes:

> “Please read my YAML file and create the Pod.”

### Breakdown:

- `kubectl` = Tool to talk to Kubernetes 🗣️
- `apply` = Create or update resources
- `-f` = File
- `grade-submission-portal-pod.yaml` = Your instruction file

### How It Works (Step-by-Step)

1. kubectl reads the YAML file
2. Sends instructions to Kubernetes cluster
3. Kubernetes creates the Pod
4. Pod starts running your container (app)

### Real Life Example 🏫

Like giving a project file to a teacher:

> “Please build this model exactly as written.”

---

# 🔍 2. `kubectl describe grade-submission-portal`

## ⚠️ Important Note

This command is actually **incomplete** and may cause error.

Because Kubernetes needs:

- Resource type (pod, service, deployment)

Correct command should be:

```bash
kubectl describe pod grade-submission-portal
```

### Simple Meaning (If Used Correctly)

It shows full details about the Pod.

Think of it like:

> A report card of your Pod 📄

---

# 📊 3. `kubectl describe pod grade-submission-portal`

## 🌟 Simple Meaning

This command tells Kubernetes:

> “Show me complete details of my Pod.”

### What Information It Shows:

- Pod status (Running / Pending / Failed)
- Container image
- CPU & Memory settings
- Events (errors if any)
- Node where Pod is running
- Ports and configuration

---

# ⚙️ How These Commands Work Together (Full Workflow)

## 🪜 Step 1: Create the Pod

```bash
kubectl apply -f .\grade-submission-portal-pod.yaml
```

Kubernetes reads YAML → Creates Pod 📦

---

## 🪜 Step 2: Check Pod Details

```bash
kubectl describe pod grade-submission-portal
```

Kubernetes shows:

- Is Pod running?
- Any error?
- Resource usage?
- Container info?

---

# 🧠 What Happens Internally (Very Easy Flow)

Your System → kubectl → Kubernetes Cluster → Pod Created → Container Runs App

Like:
Student → School Office → Principal → Classroom Setup 🎓

---

# ❓ Why We Use `describe` After `apply`?

Because after creating a Pod, we must check:

- Did it start successfully?
- Is the image pulled correctly?
- Any crash or error?

If something fails, `describe` shows the exact reason.

---

# 🧪 Bonus (Very Useful Command for Beginners)

To quickly check Pod status:

```bash
kubectl get pods
```

Output example:

- Running ✅ (Good)
- Pending ⏳ (Still starting)
- CrashLoopBackOff ❌ (Error in container)

---

# 🚨 Common Mistake (Student Warning)

Wrong:

```bash
kubectl describe grade-submission-portal
```

Right:

```bash
kubectl describe pod grade-submission-portal
```

Because Kubernetes always needs the resource type.

---

# 🎓 Final Super Easy Summary

These commands together mean:

> 1️⃣ Apply → Create my Pod from YAML file
> 2️⃣ Describe → Show full details and health of the Pod

So the working flow is:

```bash
kubectl apply -f .\grade-submission-portal-pod.yaml
kubectl get pods
kubectl describe pod grade-submission-portal
```

This is the basic and most important Kubernetes workflow every DevOps and cloud student must learn 🚀
Perfect for real projects, FYP deployments, and cloud-native applications.

## 📜 Kubernetes Useful Commands (Logs & Port Forwarding)

This section explains the basic Kubernetes commands used to check logs and access the running application inside the Pod.

---

# 🪵 1. View Pod Logs

```bash
kubectl logs grade-submission-portal
```

This command displays the logs generated by the container running inside the Pod.

Logs include:

Application startup messages

Errors and warnings

Runtime output of the application

This is very useful for troubleshooting and verifying whether the application is running properly.

# 🪵 2. View Live Logs (Real-Time Monitoring)

```bash
kubectl logs -f grade-submission-portal
```

The -f flag stands for follow, which streams logs in real time as the application runs.

Use this command when:

Debugging live issues

Monitoring application behavior

Testing the application continuously

# 🪵 3. Access the Application Using Port Forwarding

```bash
kubectl port-forward grade-submission-portal 8080:5001
```

This command forwards a local port to the container port inside the Pod.

Explanation:

grade-submission-portal → Pod name

8080 → Local machine port

5001 → Container port (where the app is running)

After executing this command, the application can be accessed in the browser at:

```bash
http://localhost:8080
```

This is especially useful when the Pod is running inside a cluster without an external service.

---

🔄 Recommended Workflow

```bash
# Create the Pod
kubectl apply -f .\grade-submission-portal-pod.yaml

# Check detailed Pod status
kubectl describe pod grade-submission-portal

# View logs
kubectl logs grade-submission-portal

# Monitor logs in real-time
kubectl logs -f grade-submission-portal

# Access the application locally
kubectl port-forward grade-submission-portal 8080:5001
```

## 🗑️ Deleting Pods in Kubernetes (Cleanup Commands)

This section explains how to delete Pods in Kubernetes using different methods.  
Deleting a Pod is useful when you want to stop, remove, or recreate an application running inside the cluster.

---

### 1️⃣ Delete a Specific Pod by Name

```bash
kubectl delete pod grade-submission-portal
```

🧠 Simple Explanation

This command deletes the Pod that has the exact name grade-submission-portal.

Command breakdown:

kubectl → Tool to communicate with Kubernetes

delete → Remove a resource

pod → Resource type

grade-submission-portal → Name of the Pod

When this command runs:

Kubernetes stops the container

Removes the Pod from the cluster

Frees the allocated CPU and Memory resources

This is the most direct and commonly used way to delete a single Pod.

### 2️⃣ Delete Pod Using Labels (Selector-Based Deletion)

```bash
kubectl delete pod -l app.kubernetes.io/name=grade-submission
```

Note: The correct way to delete by label is using -l (label selector).
🧠 Simple Explanation

This command deletes all Pods that match a specific label.

In this case, it deletes every Pod that has the label:

```bash
app.kubernetes.io/name=grade-submission
```

This is very useful when:

Multiple Pods belong to the same application

You want to delete a group of related Pods at once

Managing large-scale Kubernetes applications

---

🏷️ Why Label-Based Deletion is Powerful?

Kubernetes uses labels like tags or stickers on resources.
Instead of deleting Pods one by one, you can delete them using their label group.

Example:

Pod A → label: grade-submission

Pod B → label: grade-submission

Pod C → label: frontend

Using the label selector will delete only the matching Pods.
